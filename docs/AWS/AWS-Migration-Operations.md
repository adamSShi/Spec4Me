### 1.3 資料備份策略

#### 1.3.1 EC2 Snapshot 策略

| 項目 | 配置 |
|------|------|
| 備份工具 | AWS Backup |
| 備份頻率 | 每日一次 |
| 備份時間 | 03:00 UTC+9 |
| 保留期間 | 14 天 |
| 跨區域備份 | 視需求啟用（災難復原場景） |

**備份 Policy 設計：**

```json
{
  "BackupPlanName": "EC2-Daily-Backup",
  "Rules": [
    {
      "RuleName": "DailyBackup",
      "TargetBackupVaultName": "Default",
      "ScheduleExpression": "cron(0 18 * * ? *)",
      "StartWindowMinutes": 60,
      "CompletionWindowMinutes": 180,
      "Lifecycle": {
        "DeleteAfterDays": 14
      }
    }
  ]
}
```

#### 1.3.2 RDS Backup 策略

| 項目 | 配置 |
|------|------|
| 自動備份 | 啟用 |
| 備份保留 | 14 天 |
| 備份視窗 | 03:00-04:00 UTC+9 |
| 手動快照 | 重大變更前執行 |
| Point-in-Time Recovery | 啟用 (5 分鐘粒度) |

**備份視窗選擇理由：**
- 凌晨 3 點為系統低峰期
- 避開維護視窗（04:00-05:00）
- 備份完成後再進行維護

### 1.4 災難復原策略

#### 1.4.1 DR 等級定義

| DR 等級 | 策略 | RTO | RPO | 成本 |
|--------|------|-----|-----|------|
| Tier 1 | Pilot Light | 4-8 小時 | < 1 小時 | 低 |
| Tier 2 | Warm Standby | 1-2 小時 | < 15 分鐘 | 中 |
| Tier 3 | Multi-Region Active | < 5 分鐘 | 接近 0 | 高 |

**本案建議採用 Tier 1 (Pilot Light)：**
- 於 DR Region (ap-northeast-3 Osaka) 維持最小資源
- 定期同步 AMI、RDS Snapshot、FSx Backup
- 災難發生時啟動完整環境

#### 1.4.2 Pilot Light 架構

```

│   │   EC2   │ │   RDS   │ │   FSx   │ │   S3    │              │
│   │ Running │ │ Running │ │ Running │ │ Bucket  │              │
│   └────┬────┘ └────┬────┘ └────┬────┘ └────┬────┘              │
│        │           │           │           │                    │
└────────┼───────────┼───────────┼───────────┼────────────────────┘
         │           │           │           │
         │ AMI Copy  │ Snapshot  │ Backup    │ Replication
         │           │ Copy      │ Copy      │
         ▼           ▼           ▼           ▼
┌────────────────────────────────────────────────────────────────┐
│                         DR Region                               │
│                    (ap-northeast-3 Osaka)                       │
│                                                                 │

│   │   AMI   │ │   RDS   │ │   FSx   │ │   S3    │              │
│   │ Stored  │ │Snapshot │ │ Backup  │ │ Replica │              │
```

#### 1.4.3 DR 復原程序摘要

| 步驟 | 動作 | 預估時間 |
|------|------|---------|
| 1 | 確認災難範圍與決定啟動 DR | 15 分鐘 |
| 2 | 從最新 Snapshot 還原 RDS | 30-60 分鐘 |
| 3 | 從最新 Backup 還原 FSx | 30-60 分鐘 |
| 4 | 從 AMI 啟動 EC2 執行個體 | 10-15 分鐘 |
| 5 | 驗證服務運作正常 | 30 分鐘 |
| 6 | 更新 Route 53 DNS 指向 DR | 5 分鐘 |
| 7 | 通知相關人員 | 5 分鐘 |
| **總計** | | **2-4 小時** |

### 2.3 IAM 權限原則

#### 2.3.1 角色設計

| IAM Role | 用途 | 權限範圍 |
|----------|------|---------|
| EC2-AppServer-Role | AP Server 使用 | S3 讀寫、SSM、CloudWatch |
| EC2-ArcGIS-Role | ArcGIS Server 使用 | S3 讀寫、SSM、CloudWatch |
| RDS-Monitoring-Role | RDS Enhanced Monitoring | CloudWatch Metrics |
| Lambda-Backup-Role | 備份自動化 | EC2 Snapshot、RDS Snapshot |
| Admin-Role | 管理人員 | 完整權限 (需 MFA) |
| ReadOnly-Role | 一般查看 | 僅讀取權限 |

#### 2.3.2 最小權限原則實作

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3BucketAccess",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::company-app-bucket",
        "arn:aws:s3:::company-app-bucket/*"
      ]
    },
    {
      "Sid": "AllowCloudWatchLogs",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:ap-northeast-1:*:log-group:/aws/ec2/*"
    }
  ]
}
```

#### 2.3.3 IAM 管理建議

| 項目 | 建議 |
|------|------|
| Root 帳戶 | 啟用 MFA、不使用於日常操作 |
| 管理員存取 | 使用 IAM Role + MFA，非 IAM User |
| 服務帳戶 | 使用 IAM Role，非 Access Key |
| 權限審查 | 每季審查 IAM Policy，移除未使用權限 |
| Access Key | 定期輪換 (90 天)，禁止寫入程式碼 |

### 3.3 分層架構優化建議

#### 3.3.1 短期優化（0-6 個月）

| 項目 | 建議 | 效益 |
|------|------|------|
| Session 外部化 | 使用 ElastiCache for Redis | 啟用 Auto Scaling 基礎 |
| 靜態內容 | 移至 S3 + CloudFront | 降低 AP Server 負載 |
| 快取策略 | 導入 CloudFront | 降低延遲、減少原始伺服器請求 |

#### 3.3.2 中期優化（6-12 個月）

| 項目 | 建議 | 效益 |
|------|------|------|
| API Gateway | 導入 Amazon API Gateway | API 版本管理、流量控制 |
| 容器化 | 評估 ECS/EKS 可行性 | 更彈性的擴展、資源利用率提升 |
| 無伺服器 | 適合的功能改用 Lambda | 降低閒置成本 |

#### 3.3.3 長期優化（12+ 個月）

| 項目 | 建議 | 效益 |
|------|------|------|
| 微服務化 | 逐步拆分單體應用 | 獨立擴展、獨立部署 |
| Multi-Region | 主動式多區域部署 | 接近零 RTO |
| 事件驅動 | 導入 EventBridge + SQS | 解耦服務、提升可靠性 |

## 4. 安全強化建議

### 4.1 最小權限原則

#### 4.1.1 IAM 最小權限實作

| 原則 | 實作方式 |
|------|---------|
| 職責分離 | 不同角色使用不同 IAM Policy |
| 範圍限定 | Resource 指定具體 ARN，非 * |
| 動作限定 | Action 僅列出必要操作 |
| 條件限定 | 使用 Condition 限制 IP/時間/MFA |

### 4.2 Security Group 原則

#### 4.2.1 設計原則

| 原則 | 說明 |
|------|------|
| 白名單模式 | 僅開放必要的 Port 與來源 |
| 最小開放 | 避免 0.0.0.0/0，除非必要（如 ALB） |
| SG 參照 | 使用 SG ID 作為來源，非 IP |
| 單一職責 | 每個 SG 對應單一服務角色 |
| 文件化 | 每條規則需有說明 |

#### 4.2.2 Security Group 規則範例

| SG | Direction | Protocol | Port | Source | 說明 |
|----|-----------|----------|------|--------|------|
| sg-alb | Inbound | HTTPS | 443 | 0.0.0.0/0 | 接受外部 HTTPS |
| sg-ap | Inbound | HTTP | 80 | sg-alb | 來自 ALB |
| sg-ap | Outbound | SQL | 1433 | sg-db | 連線至 RDS |
| sg-db | Inbound | SQL | 1433 | sg-ap | 來自 AP Server |
| sg-db | Inbound | SQL | 1433 | sg-arcgis | 來自 ArcGIS |

### 4.3 私有子網部署

#### 4.3.1 部署原則

| 服務類型 | 子網類型 | 理由 |
|---------|---------|------|
| Load Balancer | Public | 需接收外部流量 |
| NAT Gateway | Public | 需 EIP |
| Application Server | Private | 不直接對外暴露 |
| Database | Private (DB Subnet) | 最高隔離層級 |
| 管理主機 | Private | 透過 SSM 存取 |

### 4.4 Bastion Host / SSM 設計

#### 4.4.1 存取方式選擇

| 方式 | 優點 | 缺點 | 建議 |
|------|------|------|------|
| Bastion Host | 傳統方式、熟悉 | 需維護、需開放 SSH/RDP Port | 不建議 |
| SSM Session Manager | 無需開放 Port、有稽核記錄 | 需安裝 Agent | **建議採用** |
| EC2 Instance Connect | 臨時金鑰、安全 | 僅支援 Linux | 不適用 |

### 4.5 額外安全建議

#### 4.5.1 資料加密

| 資料類型 | 加密方式 | 金鑰管理 |
|---------|---------|---------|
| EBS Volume | AES-256 | AWS KMS (aws/ebs) |
| RDS | AES-256 | AWS KMS (aws/rds) |
| FSx | AES-256 | AWS KMS |
| S3 | SSE-S3 | AWS 管理 |
| 傳輸中資料 | TLS 1.2+ | ACM 憑證 |

#### 4.5.2 網路安全加強

| 項目 | 建議 |

| GuardDuty | 啟用，偵測威脅活動 |

| 項目 | 建議 |
|------|------|
| CloudTrail | 啟用，記錄所有 API 呼叫 |
